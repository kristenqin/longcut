# Platform Adapter 契约

## 目标

把 YouTube、Bilibili 等平台差异隔离在输入和播放层。

主分析流程不关心平台，只消费：

- `VideoRef`
- `VideoMetadata`
- `TranscriptSegment[]`
- `PlayerBridge`

## 接口草案

```ts
type PlatformKey = "youtube" | "bilibili";

type VideoPlatformAdapter = {
  platform: PlatformKey;
  canHandle(url: string): boolean;
  parseUrl(url: string): Promise<VideoRef>;
  fetchMetadata(ref: VideoRef): Promise<VideoMetadata>;
  fetchTranscript(ref: VideoRef, options?: TranscriptOptions): Promise<TranscriptResult>;
  getEmbedConfig(ref: VideoRef): PlayerEmbedConfig;
};

type TranscriptOptions = {
  preferredLanguage?: string;
  allowAutoGenerated?: boolean;
};

type TranscriptResult = {
  segments: TranscriptSegment[];
  language?: string;
  availableLanguages?: string[];
  source: "manual" | "auto" | "ai" | "unknown";
  coverageRatio?: number;
  quality?: TranscriptQuality;
  warnings: string[];
  raw?: unknown;
};

type TranscriptQuality = {
  segmentCount: number;
  durationCoverage?: number;
  isMonotonic?: boolean;
  emptySegmentRatio?: number;
  warnings: string[];
};

type PlayerEmbedConfig = {
  kind: "iframe" | "api";
  src: string;
  canProgrammaticSeek: boolean;
};
```

## YouTube Adapter

当前项目已有能力：

- URL 解析：`extractVideoId`
- 元数据：oEmbed
- transcript：InnerTube caption tracks
- 播放：YouTube IFrame Player API

需要改造：

- `videoId` 改成平台中立 `platformVideoId`。
- `youtube_id` 存储字段改为平台复合键。
- YouTube player 迁移到 `PlayerBridge`。

## Bilibili Adapter

B 站适配至少需要：

- URL 解析：识别 `BV`、`av`、`p`。
- 元数据：获取 `aid`、`bvid`、`cid`、分 P 列表、标题、UP 主、封面、时长，并按 URL `p` 选定目标 `cid`。
- transcript：优先通过 player info 的 subtitle 列表拿 `subtitle_url`，再解析 JSON；无字幕时可走配置化 ASR fallback。
- 播放：优先使用站外 iframe player。

B 站 `VideoRef` 建议包含：

```ts
type BilibiliVideoRef = VideoRef & {
  platform: "bilibili";
  bvid: string;
  aid?: number;
  cid?: number;
  page?: number;
};
```

B 站适配器内部建议拆成：

```txt
parseUrl(url) -> PlatformVideoRef
fetchMetadata(ref) -> VideoMetadata
selectParts(metadata, policy) -> VideoPart[]
fetchSubtitleTracks(part) -> SubtitleTrack[]
fetchTranscript(track) -> TranscriptResult
```

MVP 默认只分析 URL 指定分 P；无 `p` 时分析第 1P。多 P 全量分析应作为后续显式策略，不要在 MVP 中隐式合并。

当前 MVP 实现锚点：

| 能力 | 文件 |
|---|---|
| URL 解析、metadata、subtitle transcript、ASR fallback、iframe embed | `lib/platform/bilibili-adapter.ts`、`lib/platform/bilibili-asr.ts` |
| 平台 registry | `lib/platform/registry.ts` |
| 单元测试 | `lib/__tests__/platform-registry.test.ts` |

当前实现使用 `x/web-interface/view` 和 `x/player/v2` 获取元数据与字幕。如果公开视频返回 `need_login_subtitle` 或字幕列表为空，adapter 会尝试：

```txt
x/player/playurl -> DASH audio -> Gemini ASR -> TranscriptResult(source = "ai")
```

如果 `GEMINI_API_KEY` 未配置、ASR 被关闭、音频不可用或 ASR 失败，adapter 会返回空 transcript 和 warning；API 层返回 no-credits 错误，并带 `fallbackStatus` 说明原因。WBI 签名、登录 Cookie UI、PGC/Bangumi 和互动视频不属于本轮 MVP。

## 存储键

不要继续用单一 `youtube_id`。

建议：

```txt
platform
platform_video_id
platform_part_id
canonical_url
```

唯一键：

```txt
(platform, platform_video_id, platform_part_id)
```

对于 YouTube：

```txt
platform = youtube
platform_video_id = YouTube videoId
platform_part_id = null
```

对于 Bilibili：

```txt
platform = bilibili
platform_video_id = bvid
platform_part_id = cid
```

B 站 transcript 的更细缓存键还应包含：

```txt
aid
page
subtitle_id 或 id_str
language
source
asr_provider / asr_model（仅 source = asr/ai 时）
```

原因：字幕跟随 `cid`，同一 `cid` 还可能存在多语言、人工字幕、AI 字幕或翻译字幕。只用 `bvid` 缓存 transcript 会导致分 P 视频高概率错配。
